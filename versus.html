  • ETL vs ELT

    For both the letters stand for the same thing: extract, load, and transform.

    • Extract: process of obtaining data from a source system. Data can be retrieved from an API (python process), or it can be accessed from an OLTP database.
    • Transform: process of transforming the extracted data. This includes things like changing field types and names, enhancing data, creating schemas, etc.
    • Load: process of loading transformed data into the data asset used by the end-user. This would lead to things to further analyze the data, like reporting and visualization or ML/AI tools.
    The biggest difference between the two is the ordering of the process.

    ETL

    For ETL, the data is extracted from a source, transformed, and loaded into tables for end-user usage (can be done with Python, Spark, Scala, or SQL in the data warehouse). ETL is an older method ideal for complex transformations of smaller data sets. This creates a bottleneck because you have engineers who have to transform all the data before moving it to a bigger access point for others.

    ELT

    For ELT, the data is extracted from a source, then the raw data is loaded into a data warehouse where it is transformed. From there, the warehouse is connected to analysis tools to access the transformed tables. ELT is a newer technology that provides more flexibility to analysts and is perfect for processing both structured and unstructured data.   • OLAP vs OLTP
    OLTP

    OLTP (online transaction processing) is the processing of the datasets that contain real-time execution of transactions. Also referred to as operational database.

    OLAP

    OLAP (online analytical processing) is the changing of data within a data warehouse that leads to data analysis being able to be done Also referred to as analytical database.

  • Data Lake vs Data Warehouse
    Data Lake

    Data lakes are data stores that hold raw data. This was a popular method to be used because it was a lot cheaper to store all the raw data and then pull out what you needed rather than transforming it before storing. Now as cloud technology improves data lakes are not necessarily needed.

    Data Warehouse

    Data warehouses are data stores that hold structured (or semi-structured) data. This means that in order for data to be stored in a warehouse it will have to be transformed to be in some sort of structured format, such as connected tables. Now that more companies are going the way of ELT, this means that the raw data is being dumped in the same spot as it is being transformed. This means that some modern data warehouses are both data lakes and warehouses.

  • Table: Fact vs Dimensional

    This website had a great long description and comparison of the two tables.

    The fact table contains measurements, metrics, and facts about a business process, while the dimension table is a companion to the fact table, which contains descriptive attributes to be used as query constraining.

    Fact Table

    Fact tables are at the centre of a schema and they contain general information that create links to more detailed tables. They contain things like measurement, metrics and facts about something business related.

    Dimensional Table

    Dimensional tables are at the edges of a schema and they contain more descriptive attributes of the attributes from the fact table.

  • Key: Primary vs Foreign
    Foreign key

    Foreign key is used to describe an attribute on a table that can contain repeating values to link to other tables that might contain more details about the attribute. Some facts about foreign keys are:

    • They create a relationship between two tables
    • Multiple foreign keys can be used in a table
    • Can accept null values

    Primary key

    Primary key is used to reference the link of attributes in the detailed table to the general fact table. Some facts about primary keys are:

    • Uniquely identifies each row
    • Single primary key in a table
    • Never accepts null values

  • Schema: Star vs Snowflake
    Both star and snowflake are schemas used for storing data in a data warehouse.
    Star Schema

    Star schemas have one fact table (with foreign keys) which is split into dimension tables (linked to fact table by primary keys). This schema is not normalized, meaning there are data redundancies. Here is a graphic of how a star schema would look like:



    Snowflake Schema

    Snowflake schemas have one fact table and dimensional tables, like the star schema, but it also has lookup tables. Lookup tables are secondary dimensional tables connecting to the initial dimensional table that split the primary dimensional tables until normalized (no redundancy of data).   • Package/Library vs. Module

    Packages/Libraries
    Packages are collections of modules. In order to identify it as a package, it must contain __init__.py file.

    Libraries are collections of code that have similar functions. An example would be the Pandas library. This allows us to import the library to our code and then make use of great data analysis functions without having to create the methods ourselves. Remember that you must always first import the library before being about to call on the methods of that library. The language of a library and package can be interchanged.

    Modules
    Modules are files that have functions and global variables.

    You can think of it this way: a module is a subsection of a Package.

  • Function vs. Method
    Function

    When you have a specific task that needs to be done a couple of times throughout your code, the best practice is to make a function for it. A function is a self-contained block of code, defined using the “def” keyword, which carries out a specific task, and is called by name (source). When a function is called, it will usually only operate and return a value.

    There are two main kinds of functions:

    1. Built-in function: these are the functions that come directly with the Python language.

    2. User-defined function: these are the functions that you would create in order to help you execute.

    Method

    A method in python is a self-contained block of code that carries out a specific task - like a function. However, unlike a function, it is used on an object/class. Because it is called on an object/class it can alter the object/class's data.

    How to spot a method: if it has “.” it is a method: .method() because it must be called upon by an object/class.

  • IDE vs. Code Editor
    IDE

    IDE stands for integrated development environment. The development (including coding, compiling, debugging, executing, autocomplete, libraries) can be done in a single program with an IDE.

    Code editor

    Code editor is a text editor in which code can only be edited or modified. This requires more steps to be taken by the programmer, but is less complicated especially when working on smaller-scope projects.

    As I am currently not working on large scope projects, I have chosen to use a code editor instead of IDE.

  • Python vs. Anaconda
    Python

    Python is one of the best "readable" codes and is widely used (e.g. data science, machine learning, web/desktop applications, etc.). Uses “pip” - package installer python when installing packages (as the name suggests).

    Anaconda

    The language of Anaconda is python, however, has a lot more functionality built into it compared to python. Especially when it comes to libraries, Anaconda comes with more data science/machine learning libraries directly installed. It tends to be easier to learn for non-programmers. Uses “conda” when installing packages.

    Basically, Anaconda uses python (and R) code to allow users a bit more ease of use in regards to data science and machine learning.

  • Absolute vs. Relative File Path
    Absolute Path

    The absolute path is the best to use because there are no limitations to checking where your code is located, and less chance of an error. Easiest to just specify which folder it goes into. You list the whole path starting with /.

    Example:
    /User/Folder1/Folder2/.../FinalFolder

    Relative Path

    The relative path you use when you are in a specific folder and doing smaller projects that will not move from the folder. Try to avoid using this as this can lead to errors in code if a file is ever moved.
